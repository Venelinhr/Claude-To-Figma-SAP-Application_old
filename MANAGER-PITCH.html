What this thing does Non-technical

You describe the screen you want. The system builds it in Figma using real SAP components.

Instead of hours picking parts and drawing layouts, the designer approves a quick sketch and the finished screen appears — ready to edit.

The problem: When SAP designers create a new screen, they spend hours picking the right Fiori components, checking they're used correctly, and rebuilding the layout by hand — even for screens that are 90% the same as ones that already exist.

What we built: A design assistant that takes a plain-English business requirement — or a photo of a sketch — and produces a real, editable Figma screen using genuine SAP Fiori components. In minutes, not days.

Why it matters: Every screen the assistant generates already follows SAP Fiori Design Guidelines. That means faster prototyping, fewer design-review cycles, and consistent quality across teams.

The two-fingerprint principle

The assistant never surprises the designer. At step 3 the designer sees a plain ASCII sketch and can say "remove the table, add a status column, use a worklist instead." Only after their explicit approval does anything appear in Figma. And every component that lands on the canvas is a genuine SAP Fiori Web UI Kit component — nothing invented, nothing hallucinated.

What the designer doesn't have to do

  • Pick components by hand. The system knows all 151 SAP Web UI Kit components and picks the right ones.
  • Look up Fiori guidelines. They're baked into every design decision.
  • Check accessibility. Contrast, tap targets, focus order, and status-not-by-color-alone are auto-validated.
  • Manage design tokens. Every color, font, and spacing binds to a real SAP semantic token — no raw hex codes ever.
  • Rebuild from scratch. If the requirement changes, the designer just adjusts the ASCII sketch — the Figma build regenerates.

1. Component gate — "Is this really a Fiori component?"

Every component named in the design must exist in the 151-entry SAP Web UI Kit registry. If a spec references an invented or misspelled component (e.g. "BigButton" instead of "Button"), the build stops with a clear message before anything is placed on the canvas.

  • 151 real SAP components verified against the SAP Fiori Web UI Kit library
  • 99 direct library keys (real 40-character SAP componentKeys)
  • 132 name aliases (spec drift protection — e.g. "Btn", "Breadcrumbs" both resolve correctly)

2. Composition rules — "Does this component belong here?"

Fiori has strict rules about which components can contain which others. A Table can't sit inside a Button. A Dialog title can't contain another Dialog. The system enforces these rules bidirectionally — checking BOTH parent-accepts-child AND child-accepts-parent.

  • Every registry entry declares its validParents, validChildren, mustInclude, and mustExclude
  • Violations surface as "RULE 8 Composition rule violated: X cannot be a child of Y"
  • The bidirectional check catches asymmetric bugs (e.g. Popover allows List, but List didn't allow Popover — flagged and fixed)

3. Design tokens — "No raw hex codes, ever"

Every color, font, spacing, and border in the design must reference a real SAP semantic token — not a hardcoded hex value like #0070F2. This is what keeps designs consistent across teams and future-proof against SAP theme updates.

sapButton_Emphasized_Background sapField_BorderColor sapContent_LabelColor sapPositiveElementColor sapContent_RatedColor …19 mandatory tokens

Specs with raw hex are hard-rejected at validation. Zero exceptions.

4. Accessibility — "Nobody left behind"

Four separate accessibility validators run automatically on every built screen. These aren't nice-to-haves — they're hard-blocking checks against Fiori's §7 accessibility standard.

CheckWhat it enforces
Contrast Text-to-background ratio ≥ 4.5:1 for normal text, ≥ 3:1 for large text (WCAG AA 1.4.3)
Typography hierarchy Heading order (H1→H2→H3, no skipped levels) + SAP font-size scale
Focus & tap target All interactive elements ≥ 32×32 (Compact) or 44×44 (Cozy) — thumb-friendly
Status not by color alone Semantic colors (Warning/Error/Success) must ALSO carry an icon or text label — colorblind-safe

5. Library connectivity pre-flight — "Is the SAP kit actually reachable?"

Before the first component is placed, the plugin verifies it can reach the SAP Fiori Web UI Kit library. If the library isn't connected (e.g. designer forgot to enable it in their Figma file), the build stops with a clear "Enable SAP Web UI Kit first" message. No confusing placeholder frames, no silent failures.

The bottom line

Every screen the system produces is validated against real SAP standards. When a designer says "the assistant built it, it's Fiori-compliant" — that's not a claim, it's a machine-verifiable fact. The validation results are logged and can be shown to review boards.

The Figma plugin Where the screen lands

The Figma plugin is the last mile — where the validated design spec becomes an actual editable screen in Figma. It's a single custom plugin the designer installs once and reuses forever.

What the designer sees

The plugin opens as a small panel inside Figma with a straightforward interface:

  • Paste box — a large textarea where the designer pastes the JSON spec (or drags in a .json file)
  • Library cache indicator — shows the plugin has loaded the SAP Web UI Kit variables (1,933 semantic variables cached)
  • Two buttons: Validate (runs all 5 checks) and Build Screen (renders the design)
  • Status panel — shows validation results, build progress, and any warnings or errors
  • Build info — current build version and changelog at the top

What the plugin does under the hood

When the designer clicks Build Screen, the plugin runs a well-defined sequence:

  1. Cleans the canvas — removes any previous build to prevent stacking
  2. Pre-flight check — confirms SAP Web UI Kit library is connected
  3. Walks the spec tree — top-down, honors named slots (title, content, footer)
  4. For each component — imports the real SAP library component using its verified 40-char componentKey (not a copy or approximation)
  5. Sets variants — Cozy/Compact density, semantic states (Success/Warning/Error), interaction states
  6. Applies tokens — binds fills, strokes, and text styles to the SAP shared library variables
  7. Injects content — replaces placeholder text ("Custom Button") with real spec text ("Save", "Cancel", etc.)
  8. Sets up auto-layout — real Figma auto-layout with proper spacing, padding, and growth direction
  9. Runs post-build validators — 4 accessibility checks + token linkage audit
  10. Delivers a report — how many components placed, tokens linked, any warnings, and time taken

Why "real SAP components", not copies

This is the plugin's most important guarantee. Every element the plugin places on the canvas is a real Figma component instance pointing to the SAP Fiori Web UI Kit library — same components the SAP design team maintains. When SAP updates the design system:

  • Component instances in generated screens update automatically — no re-generation needed
  • Variants, tokens, and hover/focus states stay in sync with the design system
  • Layer panels show the real SAP-branded components (purple diamond icons) — not synthesized look-alikes
  • Designers can detach, remix, and iterate on the output as if they'd built it by hand

Failure modes handled gracefully

How it works — under the hood Overview

Three connected layers do the work:

  • Knowledge layer — 151 SAP components (each with variants, tokens, accessibility rules, do/don't guidance), 9 Fiori floorplans, and a curated 52-token design-token whitelist. Every screen is measured against this knowledge.
  • Reasoning layer — Reads the requirement, picks a floorplan, chooses components, validates the composition, and produces a machine-readable design spec (JSON).
  • Build layer — A Figma plugin that reads the spec and constructs the screen using real SAP Web UI Kit component instances from the SAP-shared library.

Three new mandatory rules (2026-07-08) New

RULE 16 — Design Flexibility & User Intent

What it is: The primary objective is to produce the best possible SAP-based solution that fulfills the user's requirements — not to enforce SAP Fiori guidelines at all costs.

What changes: SAP floorplans, component compositions, and design patterns are now the default strategy, not absolute constraints. If the user provides a reference or explicitly requests something that deviates from SAP recommendations, the plugin executes it — with an honest note about the trade-off — and moves on.

When deviations are allowed: Custom layouts that better represent the business process · Hybrid layouts combining multiple SAP patterns · Alternative component hierarchies · Detaching SAP instances when necessary (detaching is a technique, not a failure)

What doesn't change: SAP design tokens, typography tokens, accessibility, and the registry gate still apply. The measure of success shifts to "how effectively does this fulfill the user's objectives?" not "how strictly does it follow SAP?"

Full doctrine: docs/DESIGN-FLEXIBILITY-DOCTRINE.md

RULE 17 — Divide-and-Conquer Analysis

What it is: A mandatory 5-phase methodology for analyzing every reference screen. The plugin never treats a screen as a single flat image.

The 5 phases:

  1. Phase 1 — Identify the Global Shell: Start with the highest-level structure (ShellBar, Navigation, App Frame). This establishes the architectural foundation.
  2. Phase 2 — Decompose the Interface: Divide the remaining screen into independent logical regions (Navigation, Master List, Detail View, Inspector, Dialog, Table, Form, Footer).
  3. Phase 3 — Analyze One Region Completely: Apply the three core principles — Reading Order (top-left → bottom-right), Divide the Screen into Sub-regions, Complete One Sub-region Before Starting the Next — with a 10-step checklist per region.
  4. Phase 4 — Recursive Decomposition: If a region is still complex, divide it further and repeat Phase 3.
  5. Phase 5 — Assemble and Verify: Only after every region is analyzed, combine them into a complete SAP application and validate parent-child relationships, nav flow, tokens, and accessibility.

Mandatory output before every spec: A labeled region map like the one below (extracted from the canonical SAP LaMa reference):

Region A · Shell Bar → ShellBar(title="SAP Landscape Management", secondaryTitle="yana")
Region B · Side Navigation → SideNavigation + NavigationItem(Monitoring expanded, Activities selected)
Region C · Activities View (master ~28%) → DynamicPage → Panel(filter) + Panel("Activities(6)") → 3 row Panels
Region D · yanatest Detail (~28%) → DynamicPage → IconTabBar(Steps active) + Panel(filter) + Panel(Operation)
Region E · Validate System (~44%) → DynamicPage → Panel(filter) + Panel("Messages(10)") + 7 message cards

Canonical worked example: docs/CANONICAL-ANALYSIS-METHODOLOGY.md

RULE 18 — Spatial Reconstruction

What it is: Before any SAP component is selected, the layout must be measured from the reference. No arbitrary dimensions, spacing, or padding.

What gets measured (in order):

  • Overall screen proportions and dimensions
  • Width of each major region (expressed as % of total — e.g. 28% / 28% / 44%)
  • Margins, internal padding, gaps between components
  • Column/row distribution and alignment

How it's documented: Every spec generated from a reference must include a layout.confidence object:

"layout": {
  "sizingStrategy": "2000px; SideNav 200px; FCL 528/528/704",
  "paddingRationale": "16px internal (SAP compact); 8px gap",
  "confidence": {
    "overallWidth": "HIGH",
    "columnRatio": "MEDIUM — estimated from visual proportions",
    "rowHeight": "HIGH — SAP compact standard"
  }
}

Confidence levels: HIGH (clear pattern), MEDIUM (inferred from context), LOW (ambiguous — state the assumption). This makes every layout decision auditable instead of invisible.

Auto Layout first: Column ratios → gap values → padding → hug vs fill. Spacer frames are banned — gap values do the work instead.

Full doctrine: docs/SPATIAL-RECONSTRUCTION-METHODOLOGY.md

💬 Component-Guidance mode

What it is: a designer types "which component should I use for X?" and gets a Fiori-guideline-grounded answer with the recommended component, alternatives, and do/don't rules — no full screen build.

Where it lives: skill/SKILL.md now has a "Which Mode?" section at the top with two clear paths — Screen Building (the full pipeline) and Component Guidance (the targeted-question flow). The guidance handler is skill/agents/component-guidance.md.

MCP wiring: the guidance agent calls the local sap-fiori-guidelines MCP first (getFioriGuideline, searchGuidelines, getPattern), then the live UI5 API JSON via ui5-mcp-server, then Chrome MCP as a fallback. Offline-preferring, cheap, cited.

Sample interaction:

Designer: "I need to show 'Approved / Pending / Rejected' status on each row of my table. Which component?"

System: Use ObjectStatus.
• Purpose: Semantic status indicator with icon + text
• When to use: Table rows, list items, object headers where the state is one of Success / Warning / Error / Information
• Do: Pair each state with both a color AND an icon (colorblind-safe per WCAG)
• Don't: Use plain Text with color for status — that's status-by-color-alone, fails accessibility
• Alternatives: GenericTag if you also need a numeric value alongside the status; InfoLabel for compact categorical tags without state

Source: SAP Fiori — ObjectStatus

Why it matters: the same reasoning that produces good full screens can help designers make good component decisions in isolation — the daily "which tool for this job" question.
