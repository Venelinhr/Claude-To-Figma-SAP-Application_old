Strategy 1 — Never read code.js

Before: Reading plugin/figma-builder/code.js cost ~45k tokens per read (×2 = 90k). It is the plugin runtime — irrelevant to a use_figma build.

After: A PreToolUse hook blocks the read entirely. All needed data is in SAP_BUILD_MANIFEST.md (~2k tokens).

block-codejs-read.sh hook−45k tokens per read

Strategy 2 — VDI semantic-model cache

Before: Every build re-ran the full 8-stage VDI analysis on the same reference image — ~13.5k tokens each time.

After: SHA-1 of image → look for semantic-models/*-<sha8>.md. Cache hit loads ~700 tokens. 96% saving on repeat images.

semantic-models/ cache−96% on repeat

Strategy 3 — Single manifest file

Before: Build knowledge was scattered across 6+ files (VDI references, registry JSONs, token files) totalling 15–20k tokens per build.

After: SAP_BUILD_MANIFEST.md — one file, ~2k tokens, contains all 25 component keys, 25 token hexes, naming rules, and 3 hard rules.

SAP_BUILD_MANIFEST.md−76% vs 6-file bundle

Strategy 4 — One-shot build discipline

Before: Live iterative fixing in Figma — 8+ use_figma calls and 6+ screenshots per build. Each screenshot ~2k tokens.

After: ANALYZE → PLAN before executing. Single use_figma call + 1 screenshot. Parallel Promise.all imports (3–4× faster wall-time).

System Overview
Three layers, one workflow — each does only what it uniquely can

Layer 1 — Claude (Reasoning)

  • Parses business requirements
  • Reads visual references (images, Figma, PDFs)
  • Runs 30 mandatory rules
  • Runs the sap-visual-reading 8-stage skill
  • Selects SAP floorplan + components
  • Builds directly via Figma MCP
  • QA + self-repair passes

Layer 2 — Figma MCP (Execution)

  • Inserts real SAP kit component instances
  • importComponentSetByKeyAsync (parallel via Promise.all)
  • Sets variant properties + injects text
  • Builds Auto Layout frames (no Spacer frames)
  • Tags fills [sapToken] · text [typo:role]
  • Drops ◆ICON/ placeholders
  • One-shot build + one verification screenshot

Layer 3 — Plugin (Token Bridge)

  • The ONLY actor with teamlibrary permission
  • Reads [sapToken] tags → binds SAP variables
  • Swaps ◆ICON/ placeholders → real kit icons
  • Binds [typo:role] → SAP text styles
  • Runs all 4 §7 a11y validators
  • Reports: bound fills, icons, text styles, leaks
  • 2,391 LOC, MCP-bind-only
Why this split? — use_figma (Figma MCP) runs in a sandbox without teamlibrary permission. It returns 0 collections when querying the SAP Web UI Kit variables. Only a real Figma plugin can bind SAP token variables (theme-switchable, audit-clean). So Claude+MCP build the structure; the plugin does the one thing only it can do.
How Claude Communicates With Figma
Reading references · making decisions · executing the build

READ — How Claude sees Figma

Claude never "opens" Figma like a human. It calls MCP tools that return structured data:

  • get_design_context(nodeId) — exact CSS: padding, gap, font-size, bound token names, layer tree
  • get_screenshot(nodeId) — a rendered PNG to verify visual result
  • get_metadata(nodeId) — lightweight structure (ids, types, x/y/w/h) to navigate before deep reads
  • search_design_system(query) — find kit components + their 40-char keys

DECIDE — How Claude reasons

Between reading and building, Claude runs a disciplined decision chain:

  • Z-pattern region read → SAP region-type mapping (analysis MCP)
  • Confidence tiers ● confirmed / ○ inferred / ? ambiguous on every element
  • Floorplan scoring table (business fit + interaction fit + SAP compliance / 30)
  • Component scoring when 2 are valid — never picks silently
  • Reads SAP_BUILD_MANIFEST.md ONLY for keys/tokens (not the 45k-token plugin)

EXECUTE — How Claude writes to Figma

One use_figma call runs JavaScript inside Figma's plugin API:

  • Imports real component sets in parallel (Promise.all)
  • .defaultVariant.createInstance() → setProperties()
  • Builds Auto Layout containers, injects text
  • Names layers with [sapToken]/[typo:role]/◆ICON/ tags
  • Roots the frame ◆SAP-UNBOUND/ScreenName
  • ONE verification screenshot — no live iterating

The round-trip in one line:

get_design_context / screenshot (READ)
VDI analysis + scoring (DECIDE)
use_figma one-shot (EXECUTE)
get_screenshot (VERIFY)

MANDATORY · 8 loop-aware stages · grounds every decision in real SAP sources

No invented components. No hallucinated tokens. It is the disciplined "senior designer read" that prevents generic output and wrong floorplans. The Plugin — Token Bind Bridge
2,391 lines · the only teamlibrary actor · its exact role
The plugin's role — the one thing only it can do: The Figma MCP (use_figma) runs in a sandbox with no teamlibrary permission — it returns 0 variable collections from the SAP Web UI Kit. Only a real Figma plugin can call figma.teamLibrary.getAvailableLibraryVariableCollectionsAsync() and bind real SAP token variables (theme-switchable, audit-clean). So the plugin is NOT a builder — it is a binder. Claude+MCP build the structure and tag it; the plugin reads those tags and does the teamlibrary binding, icon swapping, text-style binding, and a11y validation that nothing else can.

Think of it this way: Claude is the architect who draws the plans. The Plugin is the only contractor with the SAP token vault keys. Neither can do the other's job.

31 Mandatory Rules
Quality contract — any violation stops generation · Hard Rules inside · you can add your own
⛔ Hard Rules — enforced on every build
HR-1 · Side padding = 32px always
paddingLeft = paddingRight = 32 on all containers. Never 48.
HR-2 · IconButtons = Type:Tertiary
All action icons (view/edit/delete/toolbar) → Type:'Tertiary'. Never Primary/Secondary for icon-only buttons.
HR-3 · Two-line stacked text = CENTER aligned
Label+value or title+subtitle stacked vertically → counterAxisAlignItems='CENTER'. Never top/bottom.
HR-4 · Dividers = stroke on parent, never native frames
node.strokeBottomWeight=1 + stroke color. Never createFrame() for 1px lines.
HR-5 · Form Factor = Compact by default
All SAP instances default to Compact. Only use Cozy if you explicitly say so.
HR-6 · Always Horizon Light — even if reference is dark
Dark hex has no SAP variable → Bind failure. Build light always. Only override if you say "dark theme".
HR-7 · Always end with direct Figma node URL
Every build ends with a validated ?node-id=850-45411 URL. Never the bare file URL.
HR-8 · Text nodes always use [typo:role] tags
Never raw fontName:{family:'72'} alone. Add [typo:label], [typo:heading] etc. to every text node name.
HR-9 · Always full horizontal width (FILL)
Every element, container and row fills its parent width. Set layoutSizingHorizontal='FILL' after appendChild. Table cells must sum to full width.
HR-10 · Auto-save all feedback — no action needed
Positive (bravo/great/nice/👍) and negative (bad/wrong/wtf/not SAP/👎) feedback is auto-saved to memory by the hook. Claude never asks "should I save this?".
⚙ Add your own rules — they take effect immediately

Add any rule to CLAUDE.md using the format below. Claude reads it at session start and enforces it on every build. You can override built-in rules or add entirely new ones.

## ⛔ MY RULE — Always use Cozy form factor
- Our product is touch-first — override HR-5, use Cozy everywhere

## ⛔ MY RULE — Side padding = 48px for this project
- Our design system uses 48px — override HR-1

## ⛔ MY RULE — English labels only
- Never use placeholder text in other languages

You can also say it in a prompt: "hard rule for this build: Cozy, 1440px, German labels" — Claude applies it immediately and saves corrections to memory automatically.

Stage 3
EXECUTE — Single use_figma call with clone-clear-repopulate
For SAP composites with slots: clone the original ref → find the slot frame → save prototypes from the ORIGINAL (not the clone) → clear all slot children → repopulate with fresh clones → setProperties. Apply ALL boolean props explicitly.
// Clone-Clear-Repopulate pattern
const ref = figma.currentPage.findOne(n => n.id === '699:37890');
const clone = ref.clone();
const slot = clone.findOne(n => n.name === '⿻ Navigation Items');

// Save prototypes from ORIGINAL (not clone) — no inherited overrides
const protoNavItem = ref.findAll(n => n.type === 'INSTANCE'
  && n.mainComponent?.name?.includes('Navigation Item'))[0]?.mainComponent;

// Clear slot, then repopulate fresh
[...slot.children].forEach(n => n.remove());
const inst = protoNavItem.createInstance();
slot.appendChild(inst);
inst.layoutSizingHorizontal = 'FILL';
inst.setProperties({
  '️ Text#283293:137': 'Overview',
  'Navigation Indicator / External Link#283218:12': true,  // shows | divider
  'Icon#328810:0': homeIconComp.id,
});

8 Consolidated Build Rules
RULE A · HIGHEST PRIORITY
Inspect Before Build — get_design_context on any working node BEFORE any use_figma code
RULE B · SAP COMPOSITES
Clone-Clear-Repopulate — clone existing → clear slot → fresh prototypes from ORIGINAL → setProperties
RULE C · FIGMA API
setProperties only on DIRECT slot children — fails silently on double-nested instances
RULE D · PROTOTYPE SOURCE
Never clone from a modified node — inherited overrides block writes silently. Always source from unmodified original.
RULE E · CONTENT vs STRUCTURE
Reference supplies content · SAP floorplan supplies structure — never copy reference layout visually
RULE F · POSITIVE FEEDBACK
"Bingo / 100% / perfect" = canonical — freeze it, document it, never revert or improve it
RULE G · FAILURES
Root cause before retry — identify WHY before attempting another fix. Silent failure = override inheritance or nesting depth issue.
RULE H · WIDTH
After content injection, resize if text truncates — SideNavigation minimum = 260px
