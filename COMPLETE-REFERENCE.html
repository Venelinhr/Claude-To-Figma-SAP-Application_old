Core principle
The SAP library already knows its own height, colors, typography, and spacing. The plugin's job is to stack components, not recreate them. Every native fallback is a documented exception. Every color is a SAP Horizon token. Every spec is validated before it becomes a screen. Nothing is ever deleted — builds are additive.

★ Maintenance · 2026-06-27

The 3-Place Component Update Rule

When adding or modifying any SAP component in this pipeline, three places must be updated in lockstep. Skipping any one creates a silent gap — either the validator rejects valid specs, or the plugin silently drops components at render time. Learned the hard way on 2026-06-27 when the freshly-added AvatarGroup rendered correctly in the plugin but got rejected at validation. Now enforced by an auto-running consistency checker.

The three places

#LayerFile / locationWhat goes here
1 Knowledge / Registry knowledge/components/registry/<Name>.json Schema-conforming entry with figmaComponentId field that declares the rendering strategy (40-char hex / alias:X / composed:Recipe / native:text)
2 Plugin runtime path plugin/figma-builder/code.js At least one of: buildTree handler (if (comp === 'X')), SAP_KEYS entry (40-char id), or KEY_MAP alias
3 Validator allow-list plugin/figma-builder/code.js — validateSpec() Component name must appear in KEY_MAP ∪ LAYOUT_CONTAINERS ∪ NATIVE_RENDERERS ∪ SKIP_SELF ∪ HORIZ_CONTAINERS or the explicit list
The decision tree
Is there a real SAP Figma instance? ├── YES → Add figmaComponentId (40-char hex) to SAP_KEYS │ KEY_MAP self-entry (or redirect if alias) │ Registry: figmaComponentId = "<40-char-hex>" │ └── NO → Variant of another SAP component? ├── YES → Registry: figmaComponentId = "alias:Target" │ KEY_MAP entry: SpecName: 'Target' │ └── NO → Custom composition needed Registry: figmaComponentId = "composed:Recipe" Plugin: add `if (comp === 'X') {...}` in buildTree Plugin: add 'X' to NATIVE_RENDERERS set

The consistency checker

build/check-component-consistency.py scans all three layers and reports any component that's in only some of them. It runs automatically after every build-registry-bundle.js — a non-zero exit means the build is logically broken even if syntax passes.

Before this rule was enforced, the registry could claim "95 components covered" while 26 of them silently fell through the cracks at runtime. Tests and surface-level checks passed; only end-to-end user clicks surfaced the bugs. Why 7 separate MCPs (and not just one big MCP)?
Short answer: Unix philosophy — one tool, one job, composable. A single monolithic "SAP-everything" MCP would be easier to install (one URL) but harder in every other dimension. The 7-MCP layout makes specific architectural tradeoffs:

1. Independent failure surface. If sap-fiori-guidelines crashes (bad cache file, missing dep), the other 6 keep working. A single MCP would take everything down on one bad code path. With 7, a degraded session still has 6 of 7 capabilities.

2. Different consumer models. figma is official Figma code. chrome-devtools is official Google code. ui5-mcp-server and fundamental-styles are SAP's official servers. Four of the seven are not ours to merge — even if we wanted to. Forcing them into one monolith would mean forking four official servers and maintaining the forks forever.

3. Different update cadences. Figma releases protocol changes monthly. SAP UI5 ships a new minor every quarter. Our internal registry changes per session. A monolith would force-couple all release cycles. Seven MCPs let each move independently.

4. Different security/permission scopes. figma needs Figma OAuth. chrome-devtools needs browser process spawn permission. sap-figma-community reads/writes our registry files locally. ui5 hits SAP's public docs. Combining them would mean a single MCP asking for the UNION of all permissions — much more invasive on the user's machine than 7 with narrow scopes each.

5. Cache locality. Each MCP holds the cache for the data it owns: fiori-guidelines caches design guidance, figma-community caches registry-vs-library drift, application-analysis caches region-pattern vocab. Merging would either share a giant cache (cache-invalidation hell) or duplicate caches (defeating the merge).

6. Cognitive load for the AI layer. When Claude reads "use the sap-fiori-guidelines MCP", the intent is unambiguous — there's exactly one tool surface. With a monolith, every call would need additional namespacing (sap.guidelines.get vs sap.figma.search vs sap.ui5.api) and the AI has to disambiguate. Seven MCPs map cleanly to seven task types.

The tradeoff we DO pay: installation friction. Each MCP needs its own claude mcp add line. We accept this because it's a one-time cost per environment, and the architectural wins (above) recur on every session.

Could it be done with 1 MCP? Yes — but only if we owned all 4 official ones (we don't), were willing to ship a giant permission surface (we shouldn't), and accepted that one bug takes down everything (we won't). 7 MCPs is the right shape for this problem.

★ Plugin Internals · 2026-06-26

What the plugin actually does in the background

The plugin isn't just a button that runs buildScreen(). Behind the UI sits a background library scanner, a persistent client-storage cache, two component-key dispatch maps, a Variables API resolver with reverse-RGB fallback, a global enforcer that walks the built tree, four §7 accessibility validators, and an additive build placement engine. This section documents each piece — what it does, when it runs, and what the user sees when it runs.

1 · Plugin startup sequence (auto-runs on launch)

a
figma.showUI(__html__, {width:380, height:580})UI panel opens; user sees the cache indicator + scenario buttons immediately.
b
indexLibraryVariablesOnce() fires in backgroundDetached promise — does not block the UI. Posts library-cache-state messages as it progresses.
c
Check clientStorage cache (TTL 7 days)Key sapFigmaBuilder.varIndex.v1. If savedAt + 7d > now, restore index instantly — <50ms cold start.
d
Otherwise: live scan (slow path)Posts state:'scanning' to UI → orange pill. Runs getAvailableLibraryVariableCollectionsAsync + parallel getVariablesInLibraryCollectionAsync per collection.
e

7 · Global RGB→token enforcer (safety net)

After every buildScreen, the function enforceTokensAndLayout(rootNode) walks the entire built tree and rebinds any unbound SOLID fills/strokes that match a known SAP token. Steps:

  1. Build a reverse index at plugin init: for each whitelisted token, hexToRgb(token.hex) → token name (rounded to 0.001)
  2. Recursively visit every node in the built tree
  3. For each node with a SOLID fill or stroke that has NO boundVariables.color:
    • Look up the RGB in the reverse index
    • If found → call applyFill / applyStroke to bind it
    • If not found → log raw-rgb-leak exception with node name + hex
  4. Skip spacer frames (transparent, no fill to enforce)

Catches anywhere a renderer forgot to use the binding helpers AND any legacy hex value that should have mapped to a token (handles #1D2D3E, #1D2E3B, #131E29 all → sapList_TextColor).

8 · Build placement (additive mode)

Builds NEVER delete previous screens. Placement logic:

  1. Query current page for frames carrying setPluginData('sapFigmaBuilder.screen', 'true')
  2. If none → center the new screen in the current viewport
  3. If found → find the lowest frame's y + height, place new screen at lowestBottom + 120 (120px gutter)
  4. Tag the new screen with the same sentinel for future builds
  5. Zero .remove() calls anywhere in buildScreen

Effect: every build adds to a vertical stack. The user can compare iterations side-by-side or scroll back through a build history. Snapshot system + version history stays meaningful.

Architectural summary
The plugin is an imperative builder, not a renderer. It does not "render SAP screens" — it imports official SAP Web UI Kit instances, composes them in Auto Layout, injects user data, binds every fill/stroke/spacing to a real SAP Horizon variable, and validates the result. The plugin never invents colors, sizes, or typography. Everything traces to either (a) an SAP Figma instance, (b) a SAP Horizon variable binding, or (c) a documented exception in the log. The cache + enforcer + dispatch maps are what make it survive SAP republishes, disconnected libraries, and stale specs without silent failures.
Plugin Rendering · NEW 2026-06-26

★ Zero-flicker rendering

During render, SAP instances were painting at their native 1440px width for 1–2 frames before the Auto Layout FILL constraint took effect. This manifested as content text bleeding off the left edge of the content frame during build, resolving to the correct width after completion. Three fixes were applied.

Fix 1 — Configure-before-append (DynamicPageTitle)

The entire SAP DynamicPageHeader instance — title text injection, breadcrumb injection, Header Area SLOT population, Generic Tag hiding — is now done while the instance is detached (not yet in the Figma canvas tree). parentFrame.appendChild(inst) is the very last operation, so the first paint shows the fully configured state.

Fix 2 — Width constraint strip before append (all wide components)

SAP instances ship with a hardcoded minWidth=1440. Setting layoutSizingHorizontal='FILL' after append was blocked by this floor, causing a one-frame wide paint. The fix:

// BEFORE parentFrame.appendChild(inst):
try { inst.minWidth = null; } catch(e) {}
try { inst.maxWidth = null; } catch(e) {}
try { inst.resize(width, inst.height); } catch(e) {}
parentFrame.appendChild(inst);
try { inst.layoutSizingHorizontal = 'FILL'; } catch(e) {}

Applied to: DynamicPageTitle handler, FilterBar/DynamicPageHeader handler, Table handler, and the catch-all leaf handler (which places IconTabBar, ShellBar, and all other FULL_WIDTH_COMPS).

Fix 3 — Per-child size-fix in Content build loop (Option 1)

After buildTree(node, contentCol, density, contentW) returns, every newly-added child has its minWidth/maxWidth stripped, is resized to contentW, and set to layoutSizingHorizontal='FILL'. This catches any child that escaped the inner-handler fixes.

Root cause summary
The DynamicPageTitle / IconTabBar / Table were the visible culprits in the screenshot (text clipped at left, full-width bar hanging off canvas edge). All three traced back to the same pattern: append first → configure second → FILL blocked by minWidth. The fix reverses the order: configure/resize first → append → FILL. The SAP native 48px padding visible in the Figma inspector on the Title Area is the SAP library's own padding — it's correct and not added by the plugin.
